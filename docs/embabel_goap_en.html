IntroductionPresentation Objectives

Embabel is a JVM agentic framework launched by Rod Johnson, the creator of Spring. Its distinctive feature: a deterministic GOAP planner that dynamically computes an agent's execution path, instead of forcing the developer to draw the graph manually.

What this presentation covers

  • The fundamentals of GOAP (origin, concepts, OODA loop).
  • The structural difference between the classic approach (state machine, LangGraph) and the GOAP approach.
  • A complete use case — agentic processing of an automobile insurance claim — broken down into three progressive versions.
  • The finer mechanisms: static costs, dynamic costs, goal values, with their respective use cases.
  • Five execution scenarios illustrating the diversity of behaviors produced by the same code.
  • Theproduction architecture and Embabel's decisive advantages for enterprise agentic AI.

Target Audience

Architects, tech leads, and developers involved in enterprise agentic projects. Prior knowledge of Embabel is not required. Java or Spring experience makes the code easier to read but is not a blocker.

Note. This document presents a reasoned technical analysis and a recommendation. Stack choices depend on the context of each project — team, client, regulatory constraints, and application lifetime.

01GOAP Basics

GOAP stands for Goal-Oriented Action Planning. Its theoretical roots go back to STRIPS (Stanford Research Institute Problem Solver), a symbolic planner developed at SRI in 1971 for artificial intelligence research. STRIPS introduced the concepts of typed actions with preconditions and effects, and plan search through pathfinding.

GOAP is a simplification of STRIPS adapted for real time, popularized by Jeff Orkin at the MIT Media Lab for the AI of the game F.E.A.R. (Monolith, 2005). The F.E.A.R. implementation showed that a deterministic planner could produce behaviors perceived as intelligent from a catalog of typed actions, where traditional behavior trees showed their limits. Embabel takes up the same paradigm and applies it to LLM agent orchestration.

The five fundamental concepts

Concept Role In Embabel
Action A step the agent can execute, with typed preconditions and effects. Annotated method @Action
Goal Business result to achieve. A state, not a step. @AchievesGoal on a terminal action
Condition Boolean predicate on the current state, re-evaluated after each action. @Condition returning a boolean
Blackboard Shared typed memory containing all produced objects. Managed automatically by the framework
Plan Sequence of actions computed by A*, recomputed after each step. Implicit, never written by the developer

The OODA Loop — Execution Cycle

The major consequence of the OODA loop: the plan can change during execution without any imperative routing rule being written. If a condition flips or an action returns an unexpected result, the planner automatically recomposes the rest of the path.

The objective function

plan_score = goal_value − total_path_cost

The planner looks for the plan that maximizes this score. This formula is central to understanding all the mechanisms described in the rest of the presentation.

02Classic vs GOAP — The Visual Difference

The following diagram summarizes the fundamental difference between a classic approach (state machine, LangGraph) and the GOAP approach. On the left, the developer draws every edge of the graph manually. On the right, actions are declared independently and the execution graph is computed on demand by the algorithm.

Impact on scalability

State machine

Adding an action requires modifying existing edges, rereading all routing conditions, and retesting every path. Maintenance cost grows quadratically.

GOAP Embabel

Adding an action consists of creating a new annotated bean. The planner automatically integrates it into its search. No line of existing actions is modified.

03The use case: automobile insurance claim

The whole presentation is based on a realistic use case: agentic processing of an automobile insurance claim for an insurer. Each claim is unique — property damage or bodily injury, with or without a third party, variable amounts, and possible fraud signals.

The business objective

The agent must reach a decision according to one of three possible verdicts:

Auto-approval

Low-value claim, with no fraud signal, covered by the policy. Immediate compensation without human intervention.

Human review

Claim whose amount exceeds the auto-approval threshold. Transfer to a senior adjuster with the full context.

Fraud escalation

High fraud score detected during analysis. Priority transfer to the anti-fraud unit with all signals.

Approach in five progressive versions

  1. Version 1 — Minimal agent. A single linear path, no branch. Goal: introduce the Embabel syntax without complexity.
  2. Version 2 — Three terminal paths. Several @AchievesGoal with mutually exclusive preconditions. Introduction of multiple goals.
  3. Version 3 — Interchangeable actions. Several ways to produce the same result with different costs. The GOAP planner starts making economic trade-offs.
  4. Version 4 — Business complexity. Addition of conditional contextual actions (third party, bodily injury case), a loop via @State, and an amicable settlement path. Illustrates how GOAP can coexist with other control patterns.
  5. Version 5 — Operational adaptivity. Dynamic costs and values computed at runtime according to system load, client profile, and contractual constraints. The same code produces different behaviors depending on the execution context.

Domain types

The Java records that model the business entities. Each type present on the blackboard unlocks certain actions.

04Version 1 — The Minimal Agent

The simplest possible version: a single path, no branch. It introduces the Embabel syntax without additional complexity. 

@Agent(description = "Minimal automobile claim processing")
public class MinimalClaimAgent {

  @Action
  public Policy fetchPolicy(ClaimReport c, PolicyRepository repo) {
    return repo.findByClaimant(c.id());
  }

  @Action
  public DamageEstimate analyzePhotos(ClaimReport c, AI ai) {
    return ai.withDefaultLlm()
             .creating(DamageEstimate.class)
             .fromPrompt("Assess damage from photos: " + c.photoUrls());
  }

  @Action
  @AchievesGoal(description = "Claim processed")
  public ClaimDecision decide(Policy p, DamageEstimate d, ClaimReport c) {
    BigDecimal payout = d.estimatedCost().subtract(p.deductible());
    return new ClaimDecision(Decision.APPROVED, payout.max(BigDecimal.ZERO),
                              "Basic decision based on photos");
  }
}

The plan inferred by the planner

Implicit parallelization. The two intermediate actions have no type dependency between them: neither consumes the other's output. Embabel executes them in parallel automatically, without CompletableFuture or configuration.

Limitations of this version

  • All claims follow the same path, regardless of their nature.
  • No fraud detection, no third-party management, no bodily injury case.
  • Photo analysis is systematic, even for low-value claims.
  • Costs and values have no influence: only one plan is possible.

This version reproduces a classic linear workflow. It does not benefit from planning capabilities. Version 2 introduces multiple paths.

05Version 2 — Three Terminal Paths

Addition of three terminal actions @AchievesGoal with mutually exclusive preconditions. The agent can now end in three different ways depending on context. 

@Condition(name = "hasInjury")
public boolean hasInjury(ClaimReport c) { return c.hasInjury(); }

@Condition(name = "fraudSuspected")
public boolean fraudSuspected(FraudScore s) { return s.probability() > 0.6; }

@Condition(name = "amountExceedsThreshold")
public boolean exceedsThreshold(DamageEstimate d) {
    return d.estimatedCost().compareTo(BigDecimal.valueOf(5000)) > 0;
}
@Action(cost = 2.0, pre = {"!fraudSuspected", "!amountExceedsThreshold"})
@AchievesGoal(description = "Claim processed")
public ClaimDecision autoApprove(PolicyValidation v, DamageEstimate d,
                                  FraudScore s, Policy p) { ... }

@Action(cost = 12.0, pre = {"amountExceedsThreshold", "!fraudSuspected"})
@AchievesGoal(description = "Claim processed")
public ClaimDecision seniorReview(PolicyValidation v, DamageEstimate d,
                                   FraudScore s, HumanReviewService svc) { ... }

@Action(cost = 20.0, pre = {"fraudSuspected"})
@AchievesGoal(description = "Claim processed")
public ClaimDecision escalateToFraudUnit(FraudScore score, ClaimReport c,
                                          HumanReviewService svc) { ... }

Important observation about costs

The costs in this version are not differentiating. Because the preconditions are mutually exclusive, only one terminal action is eligible at any given time. The values 2.0, 12.0, and 20.0 are purely informative: they document the intent but do not participate in any trade-off. Version 3 fixes this structural problem.

Topology of version 2

For costs to become a true optimization lever, there must be several actions that produce the same type. This is precisely what version 3 proposes.

06Version 3 — Differentiating Costs

For cost to truly drive decisions, there must be several interchangeable actions : several methods producing the same result type, with different costs and preconditions. The planner then chooses between real alternatives.

Costs only differentiate where a choice exists. A mandatory action on the critical path, however expensive it may be, does not change the planner's decision — it only increases the total.

Four ways to estimate damage

Code for the four estimators

// Option A: local heuristic, fast, no LLM. Confidence ~ 0.6
@Action(cost = 1.0, pre = {"lowStakes"})
public DamageEstimate heuristicEstimate(ClaimReport c, DamageHeuristicService s) {
    return s.estimate(c);
}

// Option B: API call to the partner garage
@Action(cost = 5.0)
public DamageEstimate garagistEstimate(ClaimReport c, GarageService svc) {
    return svc.requestEstimate(c.id());
}

// Option C: photo analysis by vision model
@Action(cost = 15.0, pre = {"photosAvailable"})
public DamageEstimate visionLLMEstimate(ClaimReport c, AI ai) {
    return ai.withLlmByRole("vision-model")
             .creating(DamageEstimate.class)
             .fromPrompt("Assess damage from: " + c.photoUrls());
}

// Option D: human expertise, reserved for large claims
@Action(cost = 80.0, pre = {"expertReviewAuthorized"})
public DamageEstimate expertHumanEstimate(ClaimReport c, ExpertService svc) {
    return svc.requestExpertVisit(c.id());
}

Refinement action — "try then refine" pattern

@Condition(name = "estimateLowConfidence")
public boolean lowConf(DamageEstimate d) { return d.confidence() < 0.7; }

// Consumes a DamageEstimate, produces a new one (refined)
// Eligible only if the confidence of the previous estimate is low
@Action(cost = 5.0, pre = {"estimateLowConfidence"})
public DamageEstimate refineWithGarage(DamageEstimate existing,
                                        ClaimReport c, GarageService svc) {
    return DamageEstimate.reconcile(existing, svc.requestEstimate(c.id()));
}
Free adaptive strategy. The planner can now build heuristic → refineWithGarage → autoApprove instead of garagist → autoApprove. Refinement runs only if needed. In cases where the heuristic is already confident enough, the cost remains minimal.

The interchangeable estimators introduced by V3 allow the planner to make real economic choices. Three scenarios show how the same agent selects a different estimator depending on the context, then a fourth illustrates the pattern try then refine.

Observation — differentiating costs

For a low-value claim with few photos, the local heuristic is enough. The planner arbitrates between two eligible estimators and selects the one with the lowest cost (1.0). Cost finally becomes a true decision lever.

Without cost, the planner would have picked at random. Without interchangeability (a single estimator producing DamageEstimate), cost would have been useless. Both conditions are required for costs to drive trade-offs.

07Version 4 — Maximum Complexity

Version 3 demonstrated action interchangeability through costs. Version 4 goes further by introducing conditional contextual actions (presence of a third party, bodily injury), a conversational exchange with the client via the mode @State, and a fourth amicable settlement path. Goal: show how GOAP composes with other control patterns without losing consistency.

New domain types

public record ThirdPartyInfo(String name, String insurer,
                              boolean cooperative, BigDecimal liability) {}

public record MedicalAssessment(BigDecimal estimatedCost, Severity severity,
                                 int recoveryDays) {}

public record ClientResponse(String message, boolean documentsProvided,
                             BigDecimal counterOffer) {}

public record AmicableSettlement(BigDecimal agreedAmount,
                                 String clientSignature, LocalDateTime signedAt) {}

// Conversational state for the exchange loop with the client
@State
public record AwaitingClientDocs(List<String> requestedDocs,
                                 Instant requestedAt, int retriesLeft) {}

New conditions

@Condition(name = "involvesThirdParty")
public boolean thirdParty(ClaimReport c) {
    return c.description().toLowerCase().matches(".*(third party|other.*vehicle).*");
}

@Condition(name = "hasInjury")
public boolean injury(ClaimReport c) { return c.hasInjury(); }

@Condition(name = "thirdPartyCooperative")
public boolean cooperative(ThirdPartyInfo t) { return t.cooperative(); }

@Condition(name = "documentsMissing")
public boolean docsMissing(DamageEstimate d) {
    return d.confidence() < 0.5 && d.damagedParts().isEmpty();
}

@Condition(name = "amicableSettlementViable")
public boolean amicable(DamageEstimate d, FraudScore f, ThirdPartyInfo t) {
    return d.estimatedCost().compareTo(BigDecimal.valueOf(8000)) < 0
        && f.probability() < 0.3
        && t.cooperative();
}

Contextual actions — automatic conditional triggering

// Activates automatically as soon as the description mentions a third party
@Action(cost = 8.0, pre = {"involvesThirdParty"})
public ThirdPartyInfo extractThirdParty(ClaimReport c, AI ai) {
    return ai.withDefaultLlm().creating(ThirdPartyInfo.class)
             .fromPrompt("Extract third-party details from: " + c.description());
}

// Activates only if bodily injury is declared
@Action(cost = 3.0, pre = {"hasInjury"})
public MedicalAssessment assessMedical(ClaimReport c, AI ai) {
    return ai.withLlmByRole("medical-specialist")
             .creating(MedicalAssessment.class)
             .fromPrompt("Assess injury severity: " + c.description());
}

Conversational loop via @State

GOAP is designed for finite plans. Waiting loops ("the client did not respond, we follow up") are poorly modeled by pathfinding. Embabel allows a local switch to state machine via the pattern @State : the action returns a waiting state, control remains on that state until resolution, then hands control back to the GOAP planner. 

// If documents are missing, ask the client and wait
@Action(cost = 2.0, pre = {"documentsMissing"})
public AwaitingClientDocs requestDocsFromClient(DamageEstimate d,
                                                ClaimReport c,
                                                ClientChannel channel) {
    var requested = List.of("HD photos", "garage invoice", "accident report");
    channel.sendMessage(c.clientId(), "Please send us...");
    return new AwaitingClientDocs(requested, Instant.now(), 3);
}

// The @State handler is managed by Embabel (callbacks on events)
// When the client responds, a new ClientResponse arrives on the blackboard
@Action(cost = 1.0)
public DamageEstimate reconcileWithClientDocs(AwaitingClientDocs waiting,
                                                ClientResponse response,
                                                DamageEstimate original) {
    if (!response.documentsProvided() && waiting.retriesLeft() > 0) {
        return original;  // the loop continues, new follow-up
    }
    return DamageEstimate.enrichedFrom(original, response);
}
The key point. The GOAP planner does not know (and does not need to know) that there is a loop underneath. From the plan's point of view, requestDocsFromClient produces an AwaitingClientDocs, et reconcileWithClientDocs consumes one. The loop is local to this subgraph and managed by Embabel's state machine. GOAP and state machine coexist without stepping on each other.

07Version 4 — Amicable Settlement and Complete Topology

New path: amicable settlement between parties

When the third party is cooperative, the amount remains reasonable, and no fraud signal is detected, the agent can propose a direct amicable settlement — a new goal that short-circuits human review. 

@Action(cost = 6.0, pre = {"amicableSettlementViable"})
@AchievesGoal(description = "Claim settled amicably", value = 90.0)
public ClaimDecision proposeAmicableSettlement(DamageEstimate d,
                                                 ThirdPartyInfo tp,
                                                 Policy p,
                                                 AmicableSettlementService svc) {
    BigDecimal splitAmount = d.estimatedCost().multiply(tp.liability());
    var settlement = svc.propose(d, tp, splitAmount);
    return new ClaimDecision(Decision.AMICABLE_SETTLEMENT,
                             splitAmount, "Amicable between parties");
}

V4 now has four possible goals with distinct values and costs. Preconditions are no longer strictly exclusive — autoApprove and proposeAmicableSettlement can be eligible simultaneously. This is the context where values truly drive trade-offs.

Goal Value Preconditions Average cost
autoApprove 100 !fraud, !threshold 10-14
proposeAmicableSettlement (new) 90 amicableSettlementViable 20-25
seniorReview 60 threshold, !fraud 35-45
fraudEscalation 40 fraudSuspected 30-40

Complete topology of V4

Version 4 demonstrates that GOAP can coexist with other paradigms (state machine for loops, planning for branches) and handle real business complexity. Eleven actions, seven conditions, four goals: an equivalent workflow in an explicit state machine would easily exceed a hundred edges.

08Version 5 — Operational Adaptivity (1/2)

Previous versions use constant costs and values, fixed at compile time. V5 turns the agent into a system reactive to its environment : costs are computed at runtime based on load, budget, and calendar; values are adjusted according to client profile and SLAs. The same code then produces radically different behaviors depending on execution context.

Three levers of adaptivity

Dynamic costs

Computed on each request according to system load, LLM budget consumed on the claim, time, or date. The agent naturally avoids saturated or over-budget resources.

Dynamic values

Adjustment of goal values according to the client profile (premium, standard, at risk) and contractual constraints (SLA, overdue status, relationship history).

Multi-tenant

The same agent behaves differently depending on the end client. Company A prioritizes automation, Company B prioritizes amicable settlement, Company C systematically requires a human — a single codebase.

Contextual dynamic costs

The implementation relies on Embabel's CostFunction contract. Below, a cost function that simultaneously checks system load, consumed budget, and calendar. 

@Component
public class AdaptiveCostFunction implements CostFunction {

    @Autowired private VisionModelMetrics visionMetrics;
    @Autowired private BudgetTracker budget;
    @Autowired private ClientProfileService clients;
    @Autowired private SLATracker sla;

    @Override
    public double costOf(Action action, WorldState state) {
        ClaimReport claim = state.get(ClaimReport.class);
        ClientProfile profile = clients.getProfile(claim.clientId());
        double base = action.getDefaultCost();

        switch (action.getName()) {
            case "visionLLMEstimate" -> {
                // Progressive penalty if the vision model is saturated
                double queue = visionMetrics.queueDepth();
                double overload = queue > 50 ? queue * 0.5 : 0;
                return base + overload;
            }
            case "detectFraud" -> {
                // Switch to a lightweight detector if a lot has already been spent
                return budget.spentOnCurrentClaim(claim.id()) > 0.50
                    ? base * 0.5 : base;
            }
            case "expertHumanEstimate" -> {
                // More expensive at the end of the month (budget pressure)
                int day = LocalDate.now().getDayOfMonth();
                return day > 25 ? base * 1.5 : base;
            }
            case "heuristicEstimate" -> {
                // Reduction if the client is premium (priority to fast processing)
                return profile.isPremium() ? base * 0.5 : base;
            }
            default -> {
                return base;
            }
        }
    }
}
Observation. The agent code (actions, conditions, goals) is not modified. The only extension is this cost function observing the environment. All adaptivity logic is concentrated in one place, isolated from the business logic.

08Version 5 — Operational Adaptivity (2/2)

Dynamic values by client profile

Goal values can also be computed at runtime via @AchievesGoal(valueFunction = "..."). This allows business priorities to be modulated according to the client's context. 

@Component
public class ClientAwareValueFunction {

    @Autowired private ClientProfileService clients;
    @Autowired private SLATracker sla;

    @Bean("autoApproveValue")
    public ToDoubleFunction<ActionContext> autoApproveValue() {
        return ctx -> {
            ClaimReport c = ctx.getBlackboard().get(ClaimReport.class);
            ClientProfile p = clients.getProfile(c.clientId());
            double base = 100.0;

            // Premium: we value fast resolution
            if (p.isPremium()) base += 30;

            // SLA almost breached: urgency to resolve
            if (sla.hoursRemaining(c.id()) < 24) base += 20;

            return base;
        };
    }

    @Bean("amicableSettlementValue")
    public ToDoubleFunction<ActionContext> amicableValue() {
        return ctx -> {
            ClaimReport c = ctx.getBlackboard().get(ClaimReport.class);
            ClientProfile p = clients.getProfile(c.clientId());

            // Company B systematically prioritizes amicable settlement
            return p.tenantId().equals("COMPANY_B") ? 130.0 : 90.0;
        };
    }
}

09Goal Value Explained

The value of a goal is the counterpart to cost in the planner's objective function. It allows trade-offs between several competing goals when they are simultaneously reachable. 

plan_score = goal_value − total_path_cost

Score calculation visualization

When value matters — decision matrix

Situation Useful value Why
One goal, one path No Same score for all plans — value only shifts a constant.
One goal, several paths No Same value for all paths — the choice is based on cost alone.
Several goals, exclusive preconditions No Only one option eligible at a time — the planner does not arbitrate.
Several goals simultaneously eligible Yes Value arbitrates between them according to business priorities.
Several goals + dynamic costs Yes Automatic strategy switches according to runtime context.
At each @AchievesGoal, ask the question: "can several goals be reached simultaneously in this context?" If yes, values will arbitrate. Otherwise, they are decorative and it is preferable to leave them at 0 to avoid confusion.

10Pitfalls to Avoid and Production Architecture

Dynamic costs and their use cases were detailed in version 5. This slide covers the classic pitfalls to avoid when modeling a GOAP agent, as well as theproduction architecture typically used to deploy an Embabel agent.

Pitfalls to avoid

Pitfall Correction
Costs on mandatory actions Place differentiating costs only on actions that have alternatives.
Exclusive preconditions + different values Either loosen the preconditions, or set value = 0.
Actions that are too fine-grained (30+ step plans) Group them into "coarse" business actions. GOAP shines between 5 and 15 actions per agent.
Output types that are too specific Create shared generic types to make actions interchangeable.
Loops in the workflow Use @State (state machine) for looped portions.

11Conclusion — Why Embabel for Agentic AI

Embabel offers a structurally different approach from current agentic frameworks: deterministic GOAP planning inherited from video games, applied to LLM agent orchestration. The agent becomes a function that intelligently combines typed business actions toward a defined goal.

The five decisive advantages

1. Explainable agents

The plan is computed before execution and can be logged, audited, and validated. A contractual differentiator for regulated sectors (banking, insurance, healthcare, public sector).

2. Structural scalability

Adding an action consists of creating a new bean. No graph edge to modify. For multi-year deliveries, the difference is measured in weeks saved.

3. Free adaptivity

Dynamic costs allow the agent to change strategy according to system load, budget, and time. Native operational regulation without imperative routing code.

4. JVM type safety

Typed Java records, IntelliJ refactoring, compile-time checks. For applications living 5 years, accumulated technical debt is controlled and regressions are avoided upstream.

5. Native Spring Boot integration

90% of enterprise infrastructure delivered out of the box: Micrometer + OpenTelemetry observability, OAuth2 / mTLS security, Spring Data persistence, messaging, scheduling, health checks, actuator. No manual assembly of 12 libraries. A prototype becomes an enterprise application without months of "productionification".

Positioning — GOAP's decisive advantages

What GOAP brings
  • Plans computed, not drawn — the algorithm composes the optimal path
  • Adding actions without impact on existing actions
  • Automatic re-planning when a condition flips
  • Economic trade-offs through static and dynamic costs
  • Auditable plan before execution — response to explainability requirements
  • Implicit parallelization of independent actions
  • Adaptive strategies emergent ("try then refine" pattern)
Contraindications
  • Strictly linear workflows without alternatives — classic chaining is enough
  • Fewer than 4–5 actions per agent — cognitive overhead is not justified
  • Predominance of loops — favor the mode @State
  • Exploration without a precise goal — favor Utility AI mode
  • Team not trained in declarative patterns — prior training required

Summary

Embabel offers the first coherent vision of an agentic framework based on deterministic GOAP planning on top of the JVM. It combines three complementary ingredients: the engineering rigor of the Spring ecosystem, the adaptive intelligence of LLMs, and pathfinding-based planning inherited from STRIPS and popularized by video games. The return on investment is significant for applications that evolve over several years.

The promise "AI for your AI" describes the mechanism at the heart of the framework: using classic AI (A*, deterministic, free, explainable) to drive generative AI (non-deterministic, costly, powerful). This reconciliation of two generations of AI precisely addresses the requirements of modern enterprise applications.
